# /commit 커맨드

저장소의 변경 내용을 분석하여 일관된 형식의 커밋 메시지로 정리하고 커밋한다.
스토어 업데이트 노트 작성에 필요한 내용이 커밋 이력에서 쉽게 추출되도록 형식을 표준화한다.

---

## 실행 절차

### Step 1: 변경 내용 수집

```bash
# 스테이징된 변경사항 확인
git diff --staged --stat

# 스테이징되지 않은 변경사항 확인
git diff --stat

# 추적되지 않는 신규 파일 확인
git status --short
```

스테이징된 파일이 없으면:
1. `git diff --stat` 로 변경된 파일 전체를 사용자에게 보여준다
2. "스테이징된 파일이 없습니다. 전체 변경사항을 커밋할까요?" 확인 후
   - YES → `git add -A` 실행
   - NO  → 사용자가 직접 스테이징할 항목을 지정하도록 안내 후 대기

### Step 2: 변경 내용 분석

스테이징된 파일들의 diff를 읽어 다음을 파악한다:

```bash
git diff --staged
```

파악할 항목:
- **변경 유형**: 아래 타입 분류표에 따라 판별
- **영향 범위**: 어떤 Feature / 시스템이 영향받는가
- **변경 요약**: 한 줄로 표현 가능한 핵심 내용
- **스토어 노출 여부**: 플레이어가 체감할 수 있는 변경인가

**타입 분류 기준**:

| 타입 | 적용 기준 | 스토어 노출 |
|------|-----------|------------|
| `feat` | 새로운 기능/시스템 추가 | ✅ 대부분 |
| `fix` | 버그 수정, 비정상 동작 해결 | ✅ 대부분 |
| `balance` | 수치 조정 (데미지, 쿨타임, 가격 등) | ✅ 경우에 따라 |
| `ui` | UI 레이아웃, 텍스트, 이펙트 변경 | ✅ 대부분 |
| `perf` | 로딩 속도, 프레임, 메모리 최적화 | ✅ 경우에 따라 |
| `refactor` | 코드 구조 변경 (동작 불변, Tidy First) | ❌ |
| `chore` | 위 유형에 해당하지 않는 모든 것 | ❌ |

하나의 커밋에 **여러 유형이 섞인 경우**:
- Tidy First 원칙에 따라 `refactor`와 다른 유형은 분리 커밋을 권장
- 단, 사용자가 분리 없이 진행을 원하면 가장 대표적인 타입 하나로 처리

### Step 3: 커밋 메시지 초안 작성

아래 형식으로 커밋 메시지 초안을 작성한다.

**형식**:
```
{타입}({범위}): {제목}

{본문 — 선택, 상세 설명이 필요한 경우}
```

**제목 규칙**:
- 한국어로 작성
- 50자 이내
- 명령형 동사로 시작 (추가, 수정, 개선, 제거, 변경)
- 마침표 없음

**범위(scope) 규칙**:
- Feature명 사용 (예: `매치3엔진`, `하트시스템`, `킹스컵이벤트`)
- 여러 Feature에 걸치면 가장 대표적인 것 하나
- 공통 시스템이면 `core`, `ui`, `infra` 등 사용

**본문 규칙** (다음 중 하나라도 해당하면 작성):
- 변경 이유가 제목만으로 불명확한 경우
- `balance` 타입에서 수치 변경 내역이 있는 경우
- `fix` 타입에서 재현 조건이 있는 경우

**작성 예시**:

```
feat(하트시스템): 하트 자동 회복 기능 추가

- 최대 5개, 30분당 1개 회복
- 최대치 도달 시 회복 타이머 정지
```

```
fix(파워업): Light Ball이 장애물을 무시하는 버그 수정
```

```
balance(적유닛): 고블린 체력 및 방어력 상향 조정

- 체력: 50 → 80
- 방어력: 0 → 5
```

```
ui(레벨클리어): 클리어 화면 별 획득 애니메이션 개선
```

```
refactor(배틀서비스): BattleService 메서드 분리 (동작 불변)
```

```
chore: 미사용 에셋 정리
```

### Step 4: 사용자 확인

작성한 커밋 메시지 초안을 사용자에게 보여준다:

```
📝 커밋 메시지 초안:
─────────────────────────────
{작성한 커밋 메시지}
─────────────────────────────
이 메시지로 커밋할까요?
1) 커밋 (그대로 진행)
2) 수정 후 커밋 (수정할 내용 알려주세요)
3) 취소
```

### Step 5: 커밋 실행

사용자가 확인하면 커밋을 실행한다.

```bash
git commit -m "{제목}" -m "{본문 (있는 경우)}"
```

커밋 완료 후 결과를 출력한다:
```
✅ 커밋 완료: {커밋 해시 앞 7자리}
   {타입}({범위}): {제목}
```

## 스토어 업데이트 노트 추출 방법

이 커밋 형식을 사용하면 다음 명령으로 스토어 노출 항목만 추출할 수 있다:

```bash
# 특정 버전 이후 스토어 노출 커밋만 추출
git log v1.0.0..HEAD --oneline \
  | grep -E "^[a-f0-9]+ (feat|fix|balance|ui|perf)\("

# 타입별 분류 추출
git log --oneline | grep "^[a-f0-9]* feat("      # 새 기능
git log --oneline | grep "^[a-f0-9]* fix("       # 버그 수정
git log --oneline | grep "^[a-f0-9]* balance("   # 밸런스
git log --oneline | grep "^[a-f0-9]* ui("        # UI 변경
git log --oneline | grep "^[a-f0-9]* perf("      # 성능 개선
```

**스토어 노출 제외 타입** (`refactor`, `chore`)은 위 추출에서 자동으로 제외된다.

---

## 사용 예시

```bash
/commit
```

---

## 주의사항

- `refactor` 커밋과 `feat`/`fix` 커밋은 분리를 권장한다 (Tidy First 원칙)
- 커밋 메시지는 반드시 사용자 확인 후 실행한다 (자동 커밋 없음)
- 테스트가 실패한 상태에서는 커밋을 권장하지 않으며, 사용자에게 경고한다
