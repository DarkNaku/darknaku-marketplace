# Game Analysis Reporter

레퍼런스 게임을 분석하여 **클론 개발이 가능한 수준의 구현 명세 보고서**를 `.md` 파일로 생성하는 스킬.
MDA(Mechanics, Dynamics, Aesthetics) 프레임워크는 분석의 렌즈로 사용하며, 보고서의 본질은 "보고 바로 만들 수 있는 기획 스펙"이다.

## MDA 프레임워크 — 분석 렌즈로 활용

| 계층 | 관점 | 역할 |
|------|------|------|
| **Aesthetics** | 플레이어 | 목표: 어떤 감정을 줘야 하는가 — 모든 설계 결정의 기준 |
| **Dynamics** | 시스템 | 구조: 규칙이 실행 중 어떤 패턴을 만드는가 |
| **Mechanics** | 개발자 | 수단: 실제로 구현해야 할 규칙·데이터·시스템 |

**분석 방향**: Aesthetics(목표) → Dynamics(구조) → Mechanics(수단) 순으로 역추적
**보고서 비중**: Mechanics 구현 명세가 가장 상세해야 한다. MDA는 "왜 이렇게 만드는가"의 근거를 제공하는 역할.

### Aesthetics 8가지 유형 (LeBlanc et al.)
| 유형 | 설명 |
|------|------|
| Sensation | 감각적 쾌감 — 시청각·햅틱 만족 |
| Fantasy | 다른 세계/존재로의 몰입 |
| Narrative | 드라마, 스토리 긴장감 |
| Challenge | 장애물 극복, 숙련도 성장 |
| Fellowship | 사회적 연결, 협력/경쟁 |
| Discovery | 탐험, 새로운 발견 |
| Expression | 자기표현, 개성 |
| Submission | 루틴, 몰입형 패시브 플레이 |

## 워크플로우

### Step 1: 게임 정보 확인
게임명을 확인한다. 제공되지 않았으면 질문한다. 게임명이 있으면 바로 Step 2로 진행한다.

### Step 2: MDA 관점 웹 검색으로 자료 수집

아래 카테고리를 **순차적으로** 검색한다. 한국어·영어를 혼용하여 최대한 많은 자료를 수집한다.

**[Aesthetics 파악 — 플레이어 경험]**
1. `"{게임명}" 게임 리뷰 재미 느낌 매력`, `"{게임명}" game review fun appeal`
2. `"{게임명}" 중독성 이유 왜 재밌어`, `"{게임명}" addictive why fun analysis`
3. `"{게임명}" 아트스타일 분위기 감성`, `"{게임명}" art style atmosphere mood`

**[Dynamics 파악 — 루프 & 시스템 흐름]**
4. `"{게임명}" 게임플레이 루프 흐름`, `"{게임명}" gameplay loop session flow`
5. `"{게임명}" 피드백 시스템 보상 구조`, `"{게임명}" feedback reward structure`
6. `"{게임명}" 난이도 곡선 밸런스 설계`, `"{게임명}" difficulty curve balance design`
7. `"{게임명}" 리텐션 시스템 복귀 유인`, `"{게임명}" retention daily reward meta`

**[Mechanics 파악 — 구현 명세]**
8. `"{게임명}" 규칙 메커니즘 시스템 상세`, `"{게임명}" mechanics rules systems detail`
9. `"{게임명}" UI 화면 구성 흐름`, `"{게임명}" UI screen layout flow`
10. `"{게임명}" 재화 종류 획득 소비 경제`, `"{게임명}" currency economy earn spend`
11. `"{게임명}" 수익화 광고 인앱결제 상품`, `"{게임명}" monetization IAP ads products`
12. `"{게임명}" 콘텐츠 스테이지 레벨 언락 진행`, `"{게임명}" content stages levels unlock`
13. `"{게임명}" 소셜 길드 랭킹 이벤트`, `"{게임명}" social guild ranking event`

각 카테고리당 최소 1회, 정보 부족 시 추가 검색.
총 검색 횟수: 최소 13회, 필요시 최대 20회.

### Step 3: MDA 분석 보고서 작성
수집된 정보를 `references/report-template.md`의 표준 템플릿에 따라 작성한다.
**반드시 이 파일을 먼저 읽은 후 작성을 시작한다.**

작성 원칙:
- **Mechanics 섹션이 가장 상세해야 한다** — 수치, 규칙, 엣지 케이스, 화면 상세 포함
- 정보가 없으면 `(추정)`으로 표기하되, 유사 장르 게임 패턴을 참고해 추정값을 제시한다
- MDA 연결은 각 Mechanic 설명 끝에 `→ Dynamic`, `→ Aesthetic` 한 줄로 명시한다
- "정보 없음"만 쓰고 넘어가지 말 것 — 추정이라도 개발자가 참고할 수 있는 내용을 제공한다

### Step 4: 파일 저장 및 전달
- 파일명: `{게임명}_analysis.md` (영문, 소문자, 띄어쓰기는 `-`로)
- 저장 위치: `/mnt/user-data/outputs/`
- 저장 후 `present_files`로 사용자에게 전달

## 품질 기준

보고서가 완성되려면 다음 기준을 **모두** 충족해야 한다:

**구현 명세 기준 (가장 중요)**
- [ ] 핵심 메커니즘마다 규칙, 승리/실패 조건, 수치, 엣지 케이스가 기술됨
- [ ] UI 주요 화면이 구성 요소 수준으로 상세하게 기술됨
- [ ] 재화 획득/소비 경로가 항목별로 구체적으로 기술됨
- [ ] 밸런스 수치가 추정이라도 제시됨
- [ ] 피드백/연출이 액션별로 기술됨 (시각/사운드/햅틱)
- [ ] 광고 배치 시점과 IAP 상품 구성이 기술됨

**MDA 분석 기준**
- [ ] Aesthetics 섹션에서 주요 유형이 근거와 함께 선정됨
- [ ] Dynamics 섹션에서 코어 루프·메타 루프·피드백 루프가 명시됨
- [ ] Mechanic마다 `→ Dynamic`, `→ Aesthetic` 연결이 명시됨
- [ ] MDA 연결 분석 표가 작성됨

**공통 기준**
- [ ] 모든 섹션 작성 (정보 없으면 추정값 제시, 추정 불가 시 "직접 플레이 필요" 명시)
- [ ] 수치는 출처 또는 `(추정)` 표기
- [ ] 기술 스택(언어, 엔진) 제외
- [ ] 한국어로 작성

## 주의사항

- 검색 결과 부족 섹션: "정보 부족 — 직접 플레이 후 보완 필요"로 표기하되, 유사 장르 패턴 기반 추정값을 함께 제시
- Mechanics 상세 명세를 MDA 분석보다 우선시한다 — MDA는 각 시스템 설명의 끝에 한 줄로 연결하는 방식으로 통합
- 수익화 모델은 반드시 포함 (앱 성공의 핵심 요소)
- 추정 내용과 확인된 내용을 명확히 구분
