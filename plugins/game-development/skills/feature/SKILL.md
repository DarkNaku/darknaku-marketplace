# Feature Manager 스킬

`features.md` 파일을 단일 소스로 사용하여 개발할 기능 목록을 관리한다.

---

## 파일 구조

`features.md`는 프로젝트 루트(또는 `Docs/` 폴더)에 위치한다. 파일이 없을 경우 GRD를 참조해 자동 생성한다.

### features.md 포맷

```markdown
# Features

> 마지막 업데이트: YYYY-MM-DD

## 📋 개발 목록

- [ ] **[FEAT-001] 기능명** — 간단한 설명
- [ ] **[FEAT-002] 기능명** — 간단한 설명
- [x] **[FEAT-003] 기능명** — 간단한 설명 ✅ 완료

## 📝 메모

(선택) 추가 맥락이나 우선순위 근거 등
```

- 목록 순서 = 개발 우선순위 순서
- `[ ]` = 미완료, `[x]` = 완료
- FEAT-NNN은 자동 부여 (001부터 순번, 삭제해도 번호 재사용 안 함)
- 완료된 항목은 목록 하단으로 이동

---

## 작업별 절차

### 1. 파일 초기화 (features.md 없을 때)

1. 프로젝트 루트와 `Docs/` 폴더에서 `features.md` 탐색
2. 없으면 GRD 파일 탐색 (`Docs/*.md`, `*GRD*.md`, `*prd*.md` 패턴)
3. GRD가 있으면 GRD의 기능/요구사항 섹션을 파싱해 초기 목록 생성
4. GRD도 없으면 빈 템플릿으로 생성 후 사용자에게 기능 입력 요청
5. 생성 위치: GRD가 `Docs/`에 있으면 `Docs/features.md`, 없으면 프로젝트 루트

### 2. 기능 추가

**기본 동작**: 별도 위치 지정이 없으면 **미완료 항목 중 맨 앞(다음 개발 항목)**에 삽입

```
사용자: "로그인 기능 추가해줘"
→ 미완료 목록 맨 앞에 삽입
→ "다음 개발 항목으로 [FEAT-004] 로그인 기능을 추가했습니다."
```

위치 지정이 있으면 그에 따름:
- "맨 뒤에 추가" → 미완료 목록 맨 끝
- "N번째로 추가" → 해당 위치
- "FEAT-002 다음에" → 해당 항목 바로 뒤

### 3. 기능 삭제

- 번호(`FEAT-003`) 또는 이름으로 식별
- 삭제 전 확인: "FEAT-003 '점수 저장' 기능을 삭제할까요?"
- 삭제 후 남은 번호는 재정렬하지 않음 (번호 고유성 유지)

### 4. 기능 수정

- 이름 또는 설명 변경
- 위치(순서) 변경
- 번호는 변경하지 않음

### 5. 완료 표시

- `[ ]` → `[x]` 변경
- 완료된 항목을 목록 하단(완료 섹션)으로 이동
- "이 기능 완료", "체크해줘", "끝났어", "done" 등의 표현 인식

### 6. 목록 조회

현재 `features.md`를 읽고 아래 형식으로 요약 출력:

```
📋 개발 목록 (미완료 N개 / 완료 N개)

▶ 다음 개발: [FEAT-001] 기능명
   ...

✅ 완료: [FEAT-003] 기능명
   ...
```

---

## 실행 절차

1. **파일 탐색**: `features.md` 존재 여부 확인
2. **없으면 초기화**: GRD 탐색 → 생성
3. **작업 수행**: 추가/수정/삭제/완료/조회 중 해당 작업 실행
4. **파일 저장**: 변경사항을 `features.md`에 반영
5. **결과 보고**: 변경된 내용을 간결하게 사용자에게 알림

---

## 규칙

- FEAT 번호는 자동 부여. 현재 최대 번호 + 1
- 완료 항목은 항상 미완료 항목 아래에 위치
- 순서 = 우선순위. 위에 있을수록 먼저 개발
- 파일 수정 후 반드시 "마지막 업데이트" 날짜 갱신
- 사용자가 GRD를 언급하거나 첨부하면 해당 GRD를 참조해 기능 목록 보완 제안
