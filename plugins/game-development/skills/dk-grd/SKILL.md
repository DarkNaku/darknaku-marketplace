# Game PRD Writer

게임 분석 보고서를 인풋으로 받아, Claude Code가 구현 지시로 활용할 수 있는 PRD(제품 요구 명세서) `.md` 파일을 생성하는 스킬.

## 전제 조건

- 인풋: 게임 분석 보고서 (아래 두 가지 방식 모두 지원)
  - **파일 업로드**: `{게임명}_analysis.md` (dk-analysis 출력물)
  - **텍스트 입력**: 분석 보고서 내용을 직접 대화에 붙여넣은 경우
- 아웃풋: `{게임명}_PRD.md` 파일
- 기술 스택 독립적 (엔진/언어 언급 없음)
- Claude Code가 `/implement`, `/analysis` 등 슬래시 커맨드와 함께 사용할 수 있는 구조

## 워크플로우

### Step 1: 인풋 확인

두 가지 방식으로 분석 보고서를 받을 수 있다:

**A. 파일 업로드된 경우**
- `/mnt/user-data/uploads/` 에서 `.md` 파일을 읽는다.

**B. 텍스트로 붙여넣은 경우**
- 현재 대화에 포함된 분석 보고서 내용을 그대로 사용한다.

**인풋이 없는 경우**
- "게임 분석 보고서를 업로드하거나 내용을 붙여넣어 주세요."라고 안내한다.

### Step 2: 분석 보고서 파싱

인풋에서 다음 정보를 추출한다:
- 게임명, 장르, 핵심 재미 요소
- 핵심 메커니즘 및 규칙
- 게임 루프 구조 (코어/메타)
- UI/UX 화면 목록 및 흐름
- 재화/경제 시스템
- 수익화 모델
- MVP 우선순위 및 구현 난이도
- 리텐션 시스템

### Step 3: PRD 작성

`references/prd-template.md`를 반드시 먼저 읽은 후, 추출된 정보를 바탕으로 PRD를 작성한다.

PRD 작성 원칙:
1. **구현 지시형 문체**: "~해야 한다", "~으로 구현한다" 형태로 작성
2. **기술 독립적**: 특정 엔진/언어/라이브러리 언급 금지
3. **완결성**: 분석 보고서에 없는 정보는 장르 표준 관행으로 합리적 추론하여 채움 — 추론한 경우 `(기본값)` 표기
4. **Feature 정의**: 공식 Feature는 스프린트 플랜(섹션 7)의 `- [ ]` 항목이다. 시스템 명세(섹션 3)는 규칙/수치/상세 스펙 전용이며 체크박스를 사용하지 않는다.
5. **체크박스 위치**: `- [ ]` 체크박스는 스프린트 플랜 항목에만 사용한다. 각 항목은 독립적으로 구현 가능한 단위로 작성한다.

### Step 4: 파일 저장 및 전달
- 파일명: `{게임명}_PRD.md` (영문 소문자, 공백은 `-`)
- 저장 위치: `/mnt/user-data/outputs/`
- `present_files`로 사용자에게 전달

## 품질 기준

완성된 PRD는 다음을 충족해야 한다:
- [ ] Claude Code가 `/implement` 커맨드 실행 시 참조할 수 있는 수준의 구체성
- [ ] 시스템 명세(섹션 3)에 체크박스 없음 — 규칙/수치/스펙만 기술됨
- [ ] 스프린트 플랜(섹션 7)의 모든 항목이 `- [ ]` 형식으로 작성됨
- [ ] 스프린트 플랜이 1~2주 단위로 구성됨 (solo 개발 1~2개월 사이클 기준)
- [ ] 화면 정의가 구현자가 UI를 설계할 수 있는 수준으로 기술됨
- [ ] 수익화 플로우가 코드로 옮길 수 있는 단계별 조건으로 기술됨
