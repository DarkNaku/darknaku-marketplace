# Feature Plan

기능 요청을 받아 프로젝트 환경에 맞는 **순차적 작업 체크리스트**를 생성하는 스킬.

작업은 성격에 따라 두 가지로 구분한다.

- **코드 작업**: 반드시 **Red → Green → Refactor** 사이클을 따른다. 구현 코드보다 테스트가 항상 먼저 작성되어야 하며, 이 순서는 어떤 이유로도 바뀌지 않는다.
- **비코드 작업**: 에셋 제작, 레벨 디자인, 설정 파일 편집 등 코드를 작성하지 않는 작업. TDD 절차 없이 작업 → 확인 순서로 진행한다.

## 전제 조건

- **인풋**: 기능 요청 (자유 텍스트, GRD 일부, 또는 기능 설명)
- **컨텍스트**: `project-context.md` (docs 폴더 또는 업로드)
- **아웃풋**: `features/{FeatureName}.md` — 체크리스트 형태의 작업 명세서

## 워크플로우

### Step 1: project-context.md 존재 확인

```bash
find . -name "project-context.md" -maxdepth 3 2>/dev/null
ls /mnt/user-data/uploads/project-context.md 2>/dev/null
```

**없는 경우**: 아래를 안내하고 중단한다.

> "project-context.md가 없습니다. project-context 스킬을 먼저 실행해 주세요.
> '프로젝트 분석해줘'를 입력해 생성한 뒤, 파일을 업로드해 주세요."

**있는 경우**: 파일 전체를 읽어 다음 정보를 추출한다.
- 폴더 구조 및 파일 생성 위치 규칙
- 네이밍 규칙 (클래스, 메서드, 파일)
- 사용 중인 패키지 및 프레임워크
- 테스트 프레임워크 및 Mock 라이브러리
- 핵심 패턴 및 코딩 컨벤션
- 프로젝트 고유의 수동 작업이 필요한 항목 (에디터 설정, 에셋 생성 등)

### Step 2: 기능 요청 분석

사용자의 기능 요청을 읽고 다음을 파악한다.

- **기능의 범위**: 이 기능이 어디서 시작해서 어디서 끝나는가
- **작업 성격 분류**: 각 작업이 코드 작업인지 비코드 작업인지 구분한다
  - 코드 작업: 로직 구현, 클래스 작성, 버그 수정 등 → TDD 사이클 적용
  - 비코드 작업: 레벨 제작, 에셋 생성, 설정 편집, 데이터 입력 등 → 작업 → 확인
- **외부 의존성**: 새로운 패키지, 에셋, 설정 파일 등이 필요한가
- **프로젝트 수동 작업**: project-context.md 기준으로 사람이 직접 해야 하는 작업이 있는가

모호한 부분이 있으면 진행 전에 사용자에게 질문한다.

### Step 3: 작업 분해 원칙

**분해 기준**
- 하나의 태스크 = 하나의 명확한 결과물 (파일 생성, 테스트 통과, 에셋 완성 등)
- 태스크 하나를 완료하는 데 30분~2시간 내외가 되도록 크기 조절
- 의존 관계에 따라 순서를 정한다

**코드 작업 순서 — 절대 바뀌지 않는 원칙**

> 테스트 작성(Red) → 최소 구현(Green) → 리팩터링(Refactor)

구현 파일을 먼저 만들고 테스트를 나중에 추가하는 순서는 허용하지 않는다. 기능 단위가 여러 개면 이 사이클을 단위별로 반복 나열한다.

**비코드 작업 순서**

> 작업 수행 → 결과 확인

테스트 코드 없이 작업 항목과 확인 항목만 나열한다. (예: 레벨 100개 제작 → 레벨 100개 정상 로드 확인)

**기본 작업 순서 템플릿**

다음을 기본으로 하되, 기능 특성에 맞게 조정한다. 코드/비코드가 혼합된 기능은 각 작업 성격에 맞는 방식을 적용한다.

*코드 작업이 포함된 경우:*
1. **준비** — 데이터 모델, 인터페이스 정의
2. **[Red] 테스트 작성** → **[Green] 최소 구현** → **[Refactor] 리팩터링** *(단위별 반복)*
3. **연결 작업** — DI 등록, 외부 의존 구현체 연결 등
4. **프레젠테이션** — UI/View 연결
5. **수동 작업** *(해당하는 경우만)*
6. **통합 확인**

*비코드 작업만 있는 경우:*
1. **작업 수행** — 에셋 제작, 데이터 입력, 설정 편집 등
2. **수동 작업** *(해당하는 경우만)*
3. **확인** — 결과물이 올바른지 검증

### Step 4: 작업 명세서 작성

기능에 코드 작업과 비코드 작업이 혼합되어 있을 수 있다. 각 작업의 성격에 맞는 단계 구성을 사용한다.

**코드 작업이 포함된 경우 템플릿:**

```markdown
# {FeatureName} 작업 명세서

## 개요
{기능 한 줄 요약}

## 완료 체크
- [ ] 모든 작업 완료 (수동 작업 포함)

## 전제 조건
- {이 기능 작업 전에 완료되어야 할 것들}

## 작업 목록

### 1단계: 준비
- [ ] `{경로}/{ClassName}` 생성 — {데이터 모델 또는 인터페이스 설명}
- [ ] ...

### 2단계: [Red] 테스트 작성
- [ ] `{테스트경로}/{Name}Tests` 생성 — {검증할 동작 설명}
- [ ] ...

### 3단계: [Green] 최소 구현
- [ ] `{경로}/{Name}` 구현 — {테스트를 통과하기 위한 최소 구현}
- [ ] ...

### 4단계: [Refactor] 리팩터링
- [ ] {리팩터링 대상 및 목표} — 테스트 통과 상태 유지 확인
- [ ] ...

### 5단계: 연결 작업
- [ ] {DI 등록, 외부 구현체 연결 등 project-context.md 기준으로 작성}
- [ ] ...

### 6단계: 프레젠테이션
- [ ] `{경로}/{View 또는 UI}` 구현 — {설명}
- [ ] ...

### 7단계: 수동 작업 *(해당하는 경우만)*
- [ ] {project-context.md 기준으로 필요한 에디터·에셋·설정 작업}
- [ ] ...

### 8단계: 통합 확인
- [ ] {전체 흐름에서 확인할 항목}
- [ ] ...

## 참고 사항
{특이사항, 주의점 등}
```

**비코드 작업만 있는 경우 템플릿:**

```markdown
# {FeatureName} 작업 명세서

## 개요
{기능 한 줄 요약}

## 완료 체크
- [ ] 모든 작업 완료 (수동 작업 포함)

## 전제 조건
- {이 기능 작업 전에 완료되어야 할 것들}

## 작업 목록

### 1단계: 작업
- [ ] {작업 항목 및 대상} — {설명}
- [ ] ...

### 2단계: 수동 작업 *(해당하는 경우만)*
- [ ] {직접 수행해야 하는 에디터·설정 작업}
- [ ] ...

### 3단계: 확인
- [ ] {결과물이 올바른지 확인할 항목}
- [ ] ...

## 참고 사항
{특이사항, 주의점 등}
```

**작성 규칙**
- 각 체크박스 항목에는 반드시 **파일 경로** 또는 **작업 대상**을 명시한다
- project-context.md의 실제 네이밍 규칙, 패키지, 패턴을 그대로 적용한다
- **코드 작업**: [Red] 테스트 작성은 항상 [Green] 구현보다 앞에 온다 — 이 순서는 절대 바뀌지 않는다
- **비코드 작업**: TDD 단계 없이 작업 → 확인 순서로만 구성한다
- 코드/비코드가 혼합된 경우 각 작업 단위의 성격에 맞게 해당 섹션을 구성한다
- 해당 기능에 불필요한 단계는 생략한다
- project-context.md에 없는 패턴이나 도구는 태스크에 포함하지 않는다

### Step 5: 사용자 확인

작성된 작업 명세서 전체 내용을 사용자에게 보여주고 승인을 요청한다.

```
📋 작업 명세서: features/{FeatureName}.md
─────────────────────────────────────────
{작업 명세서 전체 내용}
─────────────────────────────────────────
이 계획으로 진행할까요?
1) 확정 (파일 저장)
2) 수정 후 확정 (수정할 내용을 알려주세요)
3) 취소
```

- **1) 확정**: Step 6으로 진행.
- **2) 수정 후 확정**: 요청한 내용을 반영한 뒤 다시 보여주고 재확인한다. 확인 완료 시 Step 6으로 진행.
- **3) 취소**: 파일을 저장하지 않고 종료한다.

### Step 6: 파일 저장 및 전달

- 저장 경로: `features/{FeatureName}.md` (영문, PascalCase)
- `present_files`로 사용자에게 전달

## 품질 기준

완성된 작업 명세서는 다음을 충족해야 한다.

- [ ] 타이틀 바로 아래 "완료 체크" 섹션에 단일 체크 항목이 포함됨
- [ ] 코드 작업은 Red→Green→Refactor 순서를 따름 (구현보다 테스트가 항상 먼저)
- [ ] 비코드 작업은 TDD 단계 없이 작업 → 확인 순서로만 구성됨
- [ ] 모든 태스크에 파일 경로 또는 작업 대상이 명시됨
- [ ] project-context.md의 네이밍 규칙과 패턴이 적용됨
- [ ] project-context.md에 없는 도구·패턴이 태스크에 포함되지 않음
- [ ] 각 태스크가 독립적으로 완료 가능한 크기임
- [ ] 개발자가 이 체크리스트만 보고 순서대로 작업할 수 있는 수준
