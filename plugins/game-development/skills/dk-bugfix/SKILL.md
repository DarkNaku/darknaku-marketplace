# DK-Bugfix

문제의 원인을 파악하고 수정하는 스킬. 코드 문제인 경우 회귀 테스트를 추가해 동일한 문제가 재발하지 않도록 한다.

## 전제 조건

- **인풋**: 아래 중 하나 이상
  - 문제 증상에 대한 설명 (텍스트)
  - 에러 메시지 또는 스택 트레이스
  - 로그 파일과 같은 문제를 확인 할 수 있는 파일
  - 문제 상황을 보여주는 스크린샷
- **아웃풋**:
  - 수정 내용 보고 (항상)
  - 회귀 방지 테스트 파일 (코드 문제인 경우)

---

## 워크플로우

### Step 1: 문제 정의

사용자의 입력에서 다음을 명확히 정리한다.

- **기대 동작**: 어떻게 동작해야 하는가
- **실제 동작**: 실제로 어떻게 동작하는가
- **재현 조건**: 언제/어떤 상황에서 발생하는가

정보가 부족하면 사용자에게 추가 질문한다.

```
❓ 문제를 더 정확히 파악하기 위해 확인이 필요합니다:
   - {추가로 필요한 정보}
```

### Step 2: 문제 원인 파악

관련 코드, 설정, 로그를 분석해 근본 원인을 찾는다.

```bash
# 관련 파일 탐색
find Assets/Scripts -name "*.cs" | xargs grep -l "{관련 키워드}" 2>/dev/null

# 에러 메시지 관련 코드 탐색
grep -r "{에러 키워드}" Assets/Scripts --include="*.cs" -l
```

분석 후 원인의 성격을 판별한다.

| 원인 유형 | 예시 | 다음 단계 |
|---|---|---|
| **코드 문제** | 로직 오류, 예외 미처리, 잘못된 계산 | Step 3 → Step 4 → Step 5 → Step 6 → Step 7 |
| **비코드 문제** | 에셋 설정, Inspector 값, 씬 구성 오류 | Step 4 → Step 6 → Step 7 |

원인 파악 후 사용자에게 알린다.

```
🔍 원인 파악 완료:
   원인 유형: {코드 문제 / 비코드 문제}
   원인 위치: {파일 경로 또는 설정 위치}
   원인 내용: {구체적인 원인 설명}
```

### Step 3: 회귀 테스트 작성 〔코드 문제인 경우만〕

수정 전에 버그를 재현하는 테스트를 먼저 작성한다. 이 테스트는 현재 **실패**해야 한다.

- 테스트 위치: `Assets/Tests/EditMode/` (Unity 런타임 불필요한 경우) 또는 `PlayMode/`
- 파일명: `{버그 관련 클래스명}Tests.cs` (기존 파일이 있으면 해당 파일에 추가)
- 테스트 메서드명: 한국어 + 언더스코어로 버그 상황을 명확히 표현

```csharp
// [Red] 버그를 재현하는 테스트 — 수정 전 반드시 실패해야 한다
[Test]
public void {버그_상황을_설명하는_테스트명}()
{
    // Arrange: 버그가 발생하는 조건 설정
    // Act: 버그를 유발하는 동작 실행
    // Assert: 올바른 동작을 기대값으로 검증
}
```

테스트 작성 후 실패 여부를 확인한다. 테스트가 통과하면 재현 조건이 잘못된 것이므로 Step 2로 돌아가 원인을 재분석한다.

### Step 4: 문제 수정

파악된 원인을 바탕으로 문제를 수정한다.

**수정 원칙**:
- 문제의 근본 원인을 수정한다. 증상만 억제하지 않는다.
- 수정 범위를 최소화한다. 관련 없는 코드는 건드리지 않는다.
- 비코드 문제(에셋, Inspector, 씬)는 수정 방법을 사용자에게 안내하고 직접 수행을 요청한다.

**비코드 문제 안내 형식**:
```
⚠️ 수동 수정 필요:
   {수정할 위치}: {수정 방법}

완료되면 "완료" 또는 "continue"를 입력해 주세요.
```

### Step 5: 수정 확인 〔코드 문제인 경우만〕

Step 3에서 작성한 회귀 테스트를 포함해 관련 테스트를 실행한다.

- Step 3의 회귀 테스트가 **통과**해야 한다 — [Green]
- 기존 테스트가 모두 **통과**해야 한다 — 수정으로 인한 사이드 이펙트 없음 확인

테스트 실패 시 원인을 분석하고 Step 4로 돌아가 수정한다.

```
✅ 테스트 통과
   - 회귀 테스트: PASS
   - 기존 테스트: PASS
```

### Step 6: 사용자에게 보고

수정 완료 후 아래 형식으로 결과를 보고한다.

```
✅ 버그 수정 완료

🔍 원인
   {원인 설명}

🔧 수정 내용
   {수정한 파일 및 변경 내용 요약}

🧪 회귀 테스트 (코드 문제인 경우)
   {추가된 테스트 파일 경로 및 테스트명}

⚠️ 참고 사항 (있는 경우)
   {추가로 알아야 할 사항}
```

### Step 7: 수정 확인 요청

보고 후 사용자에게 문제가 해결되었는지 확인한다.

```
🙋 문제가 해결되었나요? ("예" / "아니요")
```

**확인됨 (예)**:
- 스킬을 종료한다.

**확인되지 않음 (아니요)**:
- 이번 수정에서 변경한 모든 내용을 되돌린다.
  - 수정한 코드 파일을 원래 상태로 복원한다.
  - Step 3에서 추가한 회귀 테스트 파일을 삭제한다 (기존 파일에 추가한 경우 추가한 테스트 메서드만 제거한다).
- 되돌리기 완료 후 아래와 같이 알린다.

```
↩️ 수정 내용을 모두 되돌렸습니다. 원인 파악부터 다시 시작합니다.
```

- Step 2로 돌아가 원인 파악을 다시 진행한다.

---

## 품질 기준

- [ ] 증상이 아닌 근본 원인을 수정했는가
- [ ] 코드 문제의 경우 회귀 테스트가 수정 전 실패, 수정 후 통과하는가
- [ ] 기존 테스트가 모두 통과하는가
- [ ] 수정 범위가 문제와 관련된 코드에 한정되어 있는가
- [ ] 사용자가 수정 결과를 확인했는가
